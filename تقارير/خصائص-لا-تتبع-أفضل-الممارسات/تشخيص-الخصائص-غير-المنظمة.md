# تشخيص الخصائص غير المنظمة

## ملخص تنفيذي

المشروع تطور بشكل سريع، والنتيجة الحالية جيدة وظيفيًا، لكن توجد مناطق تعيق الصيانة والتوسع بسبب:
- مكونات ضخمة متعددة المسؤوليات.
- تكرار أنماط منطقية في أكثر من مكان.
- تشعب مصادر SQL لنفس الغرض.
- تباين في عقود الأخطاء بين الخدمات.

## 1) خصائص/ملفات ذات اختلاط مسؤوليات

### `src/app/components/Header.tsx`
- يجمع:
  - شريط التنقل
  - قائمة الجوال
  - Cart management
  - Global search
  - Checkout trigger
- التأثير:
  - صعوبة الاختبار
  - زيادة مخاطر الانحدار عند أي تعديل بسيط

### `src/app/components/dashboard/DashboardProducts.tsx`
- يجمع:
  - CRUD منتجات
  - إدارة فئات
  - سجل مؤرشف/محذوف
  - رفع/ضغط صور
  - notifications مخصصة
- التأثير:
  - تعقيد مرتفع
  - قابلية منخفضة لإعادة الاستخدام

### `src/app/components/dashboard/DashboardOrders.tsx`
- يجمع:
  - fetch/filter/pagination
  - تفاصيل الطلب
  - طباعة
  - حذف وتأكيد
- التأثير:
  - سلوك كثيف داخل ملف واحد

### `src/app/components/CheckoutDialog.tsx`
- يجمع:
  - validation
  - submit/API logic
  - state machine للخطوات
  - mapping أخطاء
- التأثير:
  - صعوبة الحفاظ على وضوح المنطق

## 2) تكرار/لااتساق في المنطق

- تكرار search/pagination patterns في أكثر من Dashboard component.
- اختلاف أسلوب أخطاء services:
  - `throw` في مكان
  - `return []` في مكان آخر
  - `return null` في مكان ثالث
- تكرار/تشعب SQL source:
  - `database_schema.sql`
  - `secure_checkout_rpc.sql`
  - `db/migrations/*`

## 3) مخاطر هيكلية يجب منعها

- إضافة ميزات جديدة داخل ملفات ضخمة بدل استخراج وحدة مستقلة.
- أي تعامل بيانات حساس مباشرة من UI.
- إضافة SQL logic خارج migration chain الرسمية.
- Logging بدون ضبط PII.

## 4) الإجراء التصحيحي المقترح

1. اعتماد `db/migrations` مصدر وحيد.
2. استخراج مكونات `Header`, `DashboardProducts`, `DashboardOrders`, `CheckoutDialog` إلى وحدات أصغر.
3. توحيد error contract على كل services.
4. توحيد notification system.
5. فرض قواعد CI تمنع:
   - direct supabase in UI
   - file complexity/size العالية

## 5) الحالة المستهدفة (بعد التصحيح)

- كل Feature مستقلة وقابلة للاختبار.
- طبقة بيانات موحدة.
- صلاحيات مفروضة من DB policies وليس UI.
- إضافة الميزات الجديدة أسرع وأقل مخاطرة.
