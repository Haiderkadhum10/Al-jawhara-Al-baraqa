# معيار إضافة خاصية جديدة

## 1) المبدأ العام

أي خاصية جديدة يجب أن تمر عبر 5 طبقات واضحة:

1. **Feature Scope**
2. **Domain/Use Case**
3. **Data Access (Service/RPC)**
4. **Security (RLS/AuthZ)**
5. **Testing + Documentation**

## 2) الهيكلية الموصى بها

```text
src/
  features/
    <feature-name>/
      components/
      hooks/
      schemas/
      mappers/
      index.ts
  lib/
    services/
    logger.ts
db/
  migrations/
docs/
```

## 3) قواعد إلزامية

- ممنوع استدعاء Supabase مباشرة من مكونات العرض.
- أي كتابة بيانات حساسة تكون عبر RPC أو مسار خدمة محكوم.
- يجب تعريف الصلاحية قبل تعريف الواجهة:
  - من يقرأ؟
  - من يكتب؟
  - ما حدود كل دور؟
- لا تكرار SQL logic في أكثر من ملف مصدر.

## 4) خطوات التنفيذ القياسية لأي Feature

1. صياغة Use Case وحدود الصلاحية.
2. إنشاء/تحديث migration في `db/migrations`.
3. بناء service/hook واحد واضح للاستهلاك.
4. بناء UI رفيع (Thin UI).
5. اختبار:
   - نجاح المسار
   - رفض الصلاحية
   - حالات الحواف
6. توثيق مختصر في `تقارير/الخصائص/تطويرات`.

## 5) معايير الأمان

- لا ثقة بأي قيمة مالية من العميل.
- التسعير والمخزون على الخادم فقط.
- تنقيح/إخفاء PII في logs.
- مراجعة RLS عند كل خاصية تؤثر على جداول حساسة.

## 6) معايير الصيانة

- لا ملف يتجاوز مسؤولية واحدة.
- لا ملف يتضخم بمزيج (UI + validation + data + side effects) دون فصل.
- توحيد نمط الأخطاء في كامل الـ services.

## 7) قائمة مراجعة قبل الدمج

- [ ] Migration موجودة ومجربة على staging.
- [ ] Security policies واضحة ومختبرة.
- [ ] Service contract ثابت وواضح.
- [ ] اختبارات E2E مضافة/محدثة.
- [ ] لا استخدام مباشر لـ `console.*` في منطق الأعمال.
- [ ] توثيق محدث.
