# خطة تطوير مرجعية

## الهدف

بناء مسار تطوير:
- منظم
- آمن
- قابل للتوسع
- منخفض كلفة الصيانة

## أولويات التنفيذ (مرتبة)

### P0 - توحيد مصادر الحقيقة
- اعتماد `db/migrations` كمصدر SQL وحيد.
- اعتبار `database_schema.sql` Snapshot فقط (أو أرشفته).
- توحيد اسم/عقد RPC: `create_order_secure` فقط.

### P1 - فصل المسؤوليات
- منع أي `supabase` مباشر داخل UI Components.
- إبقاء الوصول للبيانات داخل services/domain hooks فقط.
- منع منطق الأعمال داخل مكونات العرض الكبيرة.

### P2 - توحيد عقود الأخطاء
- توحيد أسلوب الخدمات:
  - إما `throw typed error`
  - أو `Result<T, E>`
- لا مزج بين `[]` / `null` / throw في نفس النطاق.

### P3 - تحسين نموذج البيانات
- توحيد نوع `Product.price` ليكون `number` دومًا.
- حصر `formatPrice` و`parsePrice` في طبقة العرض فقط.

### P4 - إعادة تنظيم المكونات الضخمة
- تقسيم:
  - `Header.tsx`
  - `DashboardProducts.tsx`
  - `DashboardOrders.tsx`
  - `CheckoutDialog.tsx`

### P5 - جودة وتشغيل مستمر
- إضافة E2E للتفويض:
  - admin vs non-admin
  - checkout anti-tamper
- إضافة CI checks:
  - منع `supabase` import في مجلدات UI
  - حد أقصى تعقيد/حجم ملف

## Definition of Done لأي خاصية جديدة

- مسار بيانات عبر service/RPC فقط.
- سياسات RLS أو تفويض واضح.
- اختبارات unit/integration + E2E لمسار حرج.
- توثيق قصير في `تقارير` عند إضافة تدفق مؤثر.
