# خطة تنفيذ أسبوعية تفصيلية (Task-by-Task)

## الهدف

تنفيذ التحسينات المعمارية والأمنية والتنظيمية بشكل تدريجي ومنخفض المخاطر، مع الحفاظ على استقرار المنتج.

## طريقة التنفيذ

- التنفيذ على فروع صغيرة لكل محور.
- كل مهمة لها:
  - نطاق واضح
  - ملفات مستهدفة
  - معيار قبول
  - اختبار تحقق

---

## الأسبوع 1: توحيد مصادر الحقيقة + إغلاق التشعبات الحرجة

### المهمة 1.1: اعتماد SQL source واحد رسمي
- **الوصف:** تثبيت `db/migrations` كمصدر SQL وحيد.
- **الملفات:**
  - `db/migrations/*`
  - `database_schema.sql`
  - `secure_checkout_rpc.sql`
- **الإجراء:**
  - إبقاء `db/migrations` هو المصدر التنفيذي الوحيد.
  - تحويل `database_schema.sql` إلى snapshot توثيقي فقط أو أرشفته.
  - إيقاف أي مسار موازٍ لتعريف RPC خارج migrations.
- **معيار القبول:**
  - لا توجد تعريفات SQL فعالة خارج `db/migrations`.
  - توثيق واضح في README.

### المهمة 1.2: توحيد عقد RPC للطلب
- **الوصف:** تثبيت اسم/توقيع RPC واحد فقط للطلب.
- **الملفات:**
  - `src/lib/services/checkoutService.ts`
  - `db/migrations/20260321_phase1_security_up.sql`
  - `تقارير/الخصائص/تطويرات/خريطة-الارتباطات-الكاملة.md`
- **معيار القبول:**
  - جميع الاستدعاءات تستخدم `create_order_secure` فقط.
  - لا وجود لأي اسم RPC بديل.

### المهمة 1.3: تنظيف ازدواجية المسارات والآثار الجانبية
- **الوصف:** إزالة أي تعارضات path style أو ملفات مولدة غير لازمة في git.
- **الملفات:**
  - `test-results/*` (تنظيف مخرجات مؤقتة)
  - إعدادات CI/ignore حسب الحاجة
- **معيار القبول:**
  - الشجرة نظيفة من artifacts غير المطلوبة في المستودع.

## الأسبوع 2: فصل المسؤوليات في المكونات الضخمة

- **معيار القبول:**
  - لا توجد مسؤوليات متضاربة داخل ملف واحد كبير.

### المهمة 2.3: تفكيك `DashboardOrders.tsx` و`CheckoutDialog.tsx`
- **الوصف:** فصل dialogs/print/pagination/fetch hooks.

## الأسبوع 3: توحيد طبقة البيانات وعقود الأخطاء

### المهمة 3.1: منع Supabase المباشر في UI
- **الوصف:** كل استهلاك بيانات يمر عبر services/hooks.
- **الملفات:**
  - `src/app/**/*`
  - `src/lib/services/*`
- **معيار القبول:**
  - لا import مباشر لـ `supabase` داخل مكونات العرض.

### المهمة 3.2: توحيد Error Contract
- **الوصف:** اعتماد نمط موحد للخدمات.
- **النهج المقترح:** `throw typed errors` مع mapper موحد.
- **الملفات:**
  - `src/lib/services/productsService.ts`
  - `src/lib/services/ordersService.ts`
  - `src/lib/services/customersService.ts`
  - `src/lib/services/storeSettingsService.ts`
- **معيار القبول:**
  - جميع الخدمات تتبع نفس العقد.
  - المعالجات في UI تصبح موحدة.

### المهمة 3.3: توحيد نماذج الدومين
- **الوصف:** تثبيت أنواع البيانات (خصوصًا السعر).
- **الملفات:**
  - `src/types/product.ts`
  - `src/lib/utils.ts`
  - الصفحات/المكونات المرتبطة بالعرض
- **معيار القبول:**
  - `price` عدد داخل الدومين، والتنسيق فقط في طبقة العرض.

## الأسبوع 4: جودة، CI، واختبارات التفويض

### المهمة 4.1: اختبارات E2E للصلاحيات
- **الوصف:** إضافة سيناريوات:
  - non-admin cannot manage dashboard data
  - admin can perform CRUD
  - checkout anti-tamper
- **الملفات:**
  - `tests/e2e/*`
  - `playwright.config.ts`
- **معيار القبول:**
  - اختبارات صلاحيات تمر على staging.

### المهمة 4.2: قواعد CI للصيانة
- **الوصف:** فرض حوكمة برمجية.
- **المطلوب:**
  - منع direct supabase in UI
  - حد أقصى للتعقيد/حجم الملف
  - lint + build + e2e gate
- **معيار القبول:**
  - PR لا يمر إذا خالف القواعد.

### المهمة 4.3: توثيق نهائي للتشغيل
- **الوصف:** تحديث وثائق التشغيل اليومية للفريق.
- **الملفات:**
  - `README.md`
  - `docs/architecture-security-matrix.md`
  - `تقارير/*`
- **معيار القبول:**
  - مطور جديد يستطيع فهم النظام والتطوير خلال يوم واحد.
